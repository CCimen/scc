# M011 - Architecture Convergence And Interaction Surface Slimming

## Purpose

Make SCC easier to maintain, explain, and extend before adding another runtime mode. This milestone is about convergence: one owner per concept, slimmer launch/setup/dashboard surfaces, fewer pre-production compatibility paths, clearer typed boundaries, deterministic tests, and docs that match source behavior.

S01 is inventory only. It changes no `src/**` or `tests/**` files. The S01 diff is limited to `.gsd/**` plus `AGENTS.md` milestone guidance.

## Planned slices

1. S01 - Inventory and milestone reconciliation. Code-free. Establish this file, D053, current milestone order, measured size baseline, and refactor gates.
2. S02 - Compatibility and naming cleanup. S02a resolved the marketplace/application `EffectiveConfig` naming ambiguity. Remaining S02 work is to classify facades/aliases by persisted-data protection and delete pre-production re-exports where no real migration is protected.
3. S03 - Setup wizard slimming. S03a deleted unreachable setup prompt/confirmation twins and made persistence reuse the proposed-config builder; S03b removed the now-unused `save_setup_config()` console argument; S03c moved static setup rendering blocks to `setup_ui.py` and cleared the last file-size warning; S03d extracted hook and confirmation step helpers so `run_setup_wizard()` is below the function warning threshold without changing setup behavior.
4. S04 - Dashboard and interactive launch slimming. S04a routed dashboard direct worktree start and session resume through a typed `commands.launch.resolved_workspace` helper, removed direct dashboard imports of launch internals, and kept session-history behavior unchanged. S04b moved prepared-plan completion for CLI start and dashboard-selected workspace launches into `commands.launch.completion`; S04c moved the start wizard tail to the same owner; S04d moved created-worktree auto-start out of `commands/worktree/worktree_commands.py` while preserving its worktree-specific launch tail; S04e moved interactive prompt-step handlers into `commands.launch.wizard_steps`, introduced explicit `WizardExit` handler outcomes, and tightened the hotspot guard; S04f unified quick-resume terminal outcomes on the same `WizardExit` contract; S04g extracted concrete CLI start plan/prepared-plan helpers so `flow.py:start` is below the function warning threshold; S04h split the `flow_session.py` session-selection router into concrete auto-detect/select/resume helpers while keeping the same owner and select-over-resume precedence; S04i mapped dashboard request exceptions through one typed request-to-event helper, reducing `run_dashboard()` from 233 to 64 lines.
5. S05 - Typed trust-boundary hardening. Replace high-value `Any`/dict-shaped policy or audit values with typed contracts where they cross boundaries.
6. S06 - Guardrail and test-debt cleanup. Earlier S06 work removed stale future-feature skips/xfails; S06f split `reset_cmd()` into concrete CLI phase helpers after adding command-level characterization tests, reducing it from 274 to 170 lines without moving maintenance behavior out of `scc_cli.maintenance`; S06g split org config validation into source loading, validation, and JSON/human rendering helpers, reducing `org_validate_cmd()` below the function warning threshold without changing the Typer command surface; S06h consolidated team validation warning/result branches and successful response construction, reducing `team_validate()` below the warning threshold without changing the command contract.
7. S07 - Docs truth and duplication cleanup. Update `../scc-cli-docs` after source changes and remove stale or duplicate claims.

## Non-goals

- Do not build runtime/devcontainer interoperability in M011; it is M012.
- Do not implement SSO, SCIM, SBOM, signed policy, audit export, or a project registry in M011.
- Do not create a new `EffectiveContext` stack while the effective-config owner can evolve.
- Do not introduce new plugin systems, one-implementation interfaces, broad helper modules, or compatibility layers for imagined future users.
- Do not optimize scanner findings without profiling or realistic data-size evidence.
